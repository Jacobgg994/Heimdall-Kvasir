## 🔥 สิ่งที่ได้เรียนรู้

### 1. fastapi-best-practices (zhanymkanov)
- **คืออะไร:** คลังบทเรียนและแนวปฏิบัติที่ดีที่สุดสำหรับการเขียน FastAPI ใน production — รวบรวมจากประสบการณ์จริงที่ startup (~17.6k stars) ครอบคลุมตั้งแต่โครงสร้างโปรเจกต์ไปจนถึงเทคนิคการ optimize
- **สิ่งที่ได้:**
  - **โครงสร้างแบบ Domain-Driven:** จัดโฟลเดอร์ตาม business domain (auth/, posts/, ฯลฯ) ไม่ใช่ตามชนิดไฟล์ — inspired by Netflix's Dispatch แต่ละ domain มี router.py, schemas.py, service.py, dependencies.py ของตัวเอง
  - **Async/Sync เส้นแบ่งสำคัญ:** `async def` ที่มี `time.sleep()` จะ block event loop ทั้งหมด ใช้ `def` ธรรมดาให้ FastAPI ส่งไป threadpool แทน หรือใช้ `await asyncio.sleep()` ถ้าต้อง non-blocking จริง ส่วน CPU-bound ต้องส่งไป process แยกผ่าน Celery/multiprocessing
  - **Pydantic จัดการทุกอย่าง:** ใช้ Field validators (`min_length`, `pattern`, `ge`) แทนเขียน if-check เอง สร้าง Custom BaseModel สำหรับ serialize datetime เป็นมาตรฐานเดียวกัน และแยก BaseSettings ตาม domain (AuthConfig, DBConfig) ไม่รวมไว้ในไฟล์เดียว
  - **Dependencies reuse ได้:** `Depends(valid_post_id)` ใช้ซ้ำได้หลาย endpoint, chain dependencies กันได้ (valid_post_id -> valid_owned_post), FastAPI cache ผลลัพธ์ dependencies ภายใน request เดียว
  - **SQL-First, Pydantic-Second:** Database จัดการ join และ aggregation (json_build_object) แล้วค่อย map เข้า Pydantic — ไม่เอาข้อมูลมา loop รวมใน Python
  - **Testing:** ใช้ httpx.AsyncClient + ASGITransport (`pip install httpx`) ตั้งแต่ day 1, override dependencies ผ่าน `app.dependency_overrides` ไม่ใช้ monkeypatch
  - **BackgroundTasks vs Queue:** BackgroundTasks เหมาะกับงาน <1 วินาทีที่ failure ไม่สำคัญ — ถ้า failure แล้วต้องมีคนตาม ให้ใช้ Celery/Arq/RQ ทันที
- **นำไปใช้ยังไง:** สามารถเอา domain-structured layout ไปใช้ในโปรเจกต์ FastAPI ของตัวเอง โดยเฉพาะการแยก Config และ dependencies ให้เป็น modular, และใช้เทคนิค SQL JSON aggregation เพื่อลด logic ฝั่ง Python

### 2. pure-architecture-fastapi (theseriff / Maclovi)
- **คืออะไร:** ตัวอย่างการนำ Clean Architecture (Uncle Bob) มาปรับใช้กับ FastAPI อย่างเคร่งครัด — แยก business logic ออกจาก frameworks และ tools ผ่าน Dependency Inversion
- **สิ่งที่ได้:**
  - **Dependency Inversion = หัวใจ:** dependencies ไหลเข้าหาศูนย์กลาง (จาก tools ข้างนอก เข้าหา business logic ข้างใน) ชั้นใน (entities + application) ใช้ได้แต่ Python stdlib เท่านั้น — ไม่รู้จัก FastAPI, PostgreSQL, หรือ HTTP ด้วยซ้ำ
  - **4 Layers ชัดเจน:** entities (business models) -> application (use cases) -> infrastructure (persistence, configs) -> presentation (HTTP routers) แต่ละชั้นสื่อสารผ่าน interface (abstract class / protocol) ทำให้เปลี่ยน database หรือ framework ได้โดยไม่กระทบ business logic
  - **Dishka DI Framework:** ใช้ Dishka ไม่ใช่ FastAPI Depends() สำหรับ inject dependencies — เพราะ Depends() ผูกติดกับ FastAPI ซึ่ง violation ของ Clean Architecture Dishka เป็น framework-agnostic ทำให้ use cases testable โดยไม่ต้องพึ่ง FastAPI
  - **Observability Built-in:** ใช้ asgi-monitor + OpenTelemetry สำหรับ logging (JSON format + trace context), tracing (OTLP gRPC exporter), metrics (Prometheus) — setup แยกใน bootstrap.py ให้เปิด/ปิดตาม environment
  - **Twelve-Factor Config:** config ทั้งหมดจาก environment variables ผ่าน NamedTuple — ไม่มี hardcode path/secret ใน code มี .env.dist เป็น template
  - **Production vs Test separation:** `create_app_tests()` vs `create_app_production()` — test app ไม่เปิด observability, เปิด debug mode ส่วน production app เปิด tracing/metrics ครบ
  - **API Versioning ง่าย:** `router_v1 = APIRouter(prefix="/v1")` — แค่เพิ่ม http/v2 แล้ว register router ใหม่
- **นำไปใช้ยังไง:** ใช้เป็น template สำหรับโปรเจกต์ที่ต้องการความยั่งยืนระยะยาว — architecture ที่เปลี่ยน database หรือ message broker ได้โดยไม่ต้อง rewrite business logic การแยก create_app สำหรับ test/production ก็เป็น pattern ที่เอาไปใช้ได้ทันที

### 3. Frontend-Engineering-Lab (yemisi567)
- **คืออะไร:** Open-source คลังความรู้ Frontend engineering — patterns, architecture decisions, performance techniques ที่ "usually lives in private Slack threads and senior engineers' heads" — ยังเป็น community-driven ที่เพิ่งเริ่มต้น (49 stars, หลาย section ยังว่าง)
- **สิ่งที่ได้:**
  - **Feature-Driven Architecture + Strict Colocation:** จัด structure ตาม feature ไม่ใช่ตาม tech type — `features/billing/{components/, hooks/, api/, types/}` แทนที่จะมี `components/`, `hooks/` แยก หลักการคือ "Code that changes together should live together" — เวลา feature billing เปลี่ยน ก็เปลี่ยนแค่ใน features/billing/ ไม่ต้องเดินหา 5 โฟลเดอร์
  - **Public API Boundary ผ่าน index.ts:** สิ่งที่สำคัญที่สุดคือ `index.ts` barrel file ของแต่ละ feature — export เฉพาะ public components/types ส่วน internal hooks/API clients/helpers เก็บไว้ private ถ้าไม่มี boundary นี้ คนอื่นจะ import ลึกเข้าไปใน implementation detail ทำให้ coupling สูง
  - **Enforcement with ESLint:** ใช้ `no-restricted-imports` rule ป้องกัน pattern `@/features/*/*` — บังคับให้ import ผ่าน index.ts เท่านั้น ทำให้ encapsulation แข็งแรง
  - **ไม่ต้อง overengineer:** Feature-driven architecture เหมาะกับ medium-to-large apps ที่มีหลาย contributors — ถ้าเป็น prototype หรือ simple marketing site อย่าเพิ่งใช้
  - **Testing Trophy Approach:** balance ระหว่าง unit, integration, E2E testing — ไม่ใช่แค่ testing pyramid แบบดั้งเดิม เน้น integration tests มากที่สุดเพราะให้ความคุ้มค่าสูงสุด
- **นำไปใช้ยังไง:** เอา Feature-Driven Architecture มาปรับใช้กับโปรเจกต์ FastAPI ขนาดใหญ่ก็ได้ — จริง ๆ แล้ว принцип "code that changes together lives together" ใช้กับ backend ก็ได้เหมือนกัน ส่วนแนวคิด public API boundary ผ่าน __init__.py (backend version ของ index.ts) ก็น่าสนใจ

## 🧠 สรุปสัปดาห์นี้

สาม repos นี้ cover ตั้งแต่宏观 (architecture) ไปจนถึง微观 (best practices):

| มุม | Repo | ประเด็นหลัก |
|-----|------|------------|
| **Backend Structure** | fastapi-best-practices | จัดโปรเจกต์ แยก async/sync, Pydantic mastery, SQL-first |
| **Backend Architecture** | pure-architecture-fastapi | Clean Architecture, Dependency Inversion, Dishka DI, Observability |
| **Frontend Structure** | Frontend-Engineering-Lab | Feature-driven, colocation, public API boundary |

สิ่งที่เชื่อมโยงทั้งสามคือการ **organize code โดย business domain ไม่ใช่ technical layer** — domain-driven structure ใน FastAPI, feature-driven ใน Frontend, และ clean architecture ที่แยก business logic ออกจาก infrastructure — ทั้งหมดสอน同一件事: เขียน code ให้ maintainable ในระยะยาว
