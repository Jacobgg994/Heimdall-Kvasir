### 1. zhanymkanov/fastapi-best-practices

- **คืออะไร:** คู่มือแนวปฏิบัติที่ดีที่สุดสำหรับ FastAPI ใน production รวบรวมจากประสบการณ์จริงของทีมที่รัน FastAPI มาหลายปี โดยเน้นการจัดโครงสร้างโปรเจกต์แบบโดเมน (domain-oriented) ไม่ใช่ layer-oriented

- **สิ่งที่ได้:**

  - **จัดโครงสร้างตามโดเมน ไม่ใช่ตามเลเยอร์** — แต่ละโมดูล (auth, posts, notifications) มีทุกอย่างของมันเอง: router.py, schemas.py, models.py, service.py, dependencies.py, exceptions.py, constants.py การ import ข้ามโมดูลต้องใช้ path แบบเนมสเปซชัดเจน (`from src.auth import constants as auth_constants`) ป้องกัน name collision

  - **ใช้ Dependency Injection ให้เต็มที่ — ไม่ใช่แค่ DI แต่รวม Validation และ Data Retrieval** — Dependencies ใน FastAPI สามารถใช้ validate request ที่ต้องอ่านจาก DB ได้ เช่น `valid_post_id` dependency ที่ fetch post แล้ว raise `PostNotFound` จากนั้น reuse dependency นี้ข้าม endpoint ได้ เปิดทางให้ chain dependencies (`valid_post_id` -> `parse_jwt_data` -> `valid_owned_post`) โดย FastAPI จะ cache ผลลัพธ์ภายใน request เดียวกันให้ฟรี

  - **Async ต้องทำถูกทาง** — Route แบบ sync ทำงานใน threadpool (ยังใช้ได้ถ้ามี blocking I/O) แต่ถ้าเขียน async แล้วไปใส่ blocking call เช่น `time.sleep()` จะทำให้ event loop ทั้งเส้นล็อก เทคนิคต้องใช้ `asyncio.sleep()` หรือ `run_in_threadpool` สำหรับ sync SDKs

  - **SQL-First สำหรับงานหนัก** — joins, aggregations, JSON construction ที่ซับซ้อนให้ทำใน database (PostgreSQL) โดยใช้ `json_build_object` สร้าง nested response แล้วให้ Pydantic รับช่วงต่อ ไม่ใช่มาประกอบใน Python ซึ่งช้ากว่า

  - **BackgroundTasks กับ Task Queue ต้องแยกให้ชัด** — `BackgroundTasks` ใช้ได้กับงานสั้น (< 1 วินาที) ที่ failure ไม่เป็นไร ถ้าต้อง retry, schedule, หรือ persistent queue ให้ใช้ Celery/Arq/RQ "If you'd page someone when the task is lost, it doesn't belong in BackgroundTasks"

  - **เรื่องต้องรู้ของ Serialization** — FastAPI สร้าง Pydantic object ถึง 2 รอบตอน serialize response: รอบแรกจากโค้ดเรา รอบสองจาก internal jsonable_encoder + re-validation

- **นำไปใช้ยังไง:** ใช้เป็น reference สำหรับการตั้งมาตรฐานโครงสร้างโปรเจกต์ FastAPI ของทีม — โดยเฉพาะรูปแบบ modular domain structure, dependency chaining, และ Async/sync best practices

### 2. ivan-borovets/fastapi-clean-example

- **คืออะไร:** ตัวอย่างโปรเจกต์ FastAPI ที่ใช้ Clean Architecture และ Tactical Domain-Driven Design แบบใช้งานได้จริง มี CI/CD, Docker, Auth แบบ Session-based

- **สิ่งที่ได้:**

  - **Clean Architecture 4 ชั้นชัดเจน** — โครงสร้างแบ่งเป็น `core/` (domain), `inbound/` (controllers/adapters ขาเข้า), `outbound/` (adapters ขาออก), `main/` (app wiring) แบบ port-adapter pattern โดยไม่มี framework coupling

  - **Tactical DDD + CQRS + Repository + Unit of Work** — ครบทุก pattern ที่จำเป็นสำหรับ production backend: domain entities / value objects แยกชั้น, commands/queries แยกจากกัน, repositories abstract data access, UoW จัดการ transaction boundary

  - **"No stateful globals (DI)"** — ปฏิเสธการใช้ global singletons หรือ module-level state ทุกอย่างถูก inject ผ่าน FastAPI Depends ทำให้ทดแทน dependencies ตอน test ได้ง่าย

  - **Session-based authentication + Contextual RBAC** — ใช้ session ฝั่ง server (ไม่ใช่ JWT stateless) ร่วมกับ RBAC ที่พิจารณาบริบทของ request ไม่ใช่แค่ static role claim

  - **Tooling สมัยใหม่** — ใช้ `uv` แทน pip/poetry (เร็วมาก), pre-commit hooks, Makefile, Docker Compose, Alembic สำหรับ migration

- **นำไปใช้ยังไง:** ใช้เป็น template สำหรับโปรเจกต์ Production FastAPI ของทีม — โดยเฉพาะรูปแบบ Clean Architecture + DDD + CQRS ที่ผสานกันอย่างลงตัว

### 3. gwynz/fastapi-clean-architecture (fork จาก ivan-borovets)

- **คืออะไร:** Fork ของ fastapi-clean-example ที่ถูกพัฒนาเพิ่มเติม แก้ pain point เดิมและเพิ่มรายละเอียด Clean Architecture มากขึ้น โดยใช้ Dishka DI แทน FastAPI Depends

- **สิ่งที่ได้:**

  - **โครงสร้าง Clean Architecture ที่ละเอียดกว่า** — ชั้นแบ่งเป็น:
    - `domain/` — entities, value_objects, services, enums, exceptions, ports อิสระจากทุกอย่าง
    - `application/` — commands (write), queries (read) แยกกัน, ใช้ interactors (stateless) ที่ orchestrate domain logic
    - `infrastructure/` — adapters ของจริง (SQLAlchemy, auth session handlers)
    - `presentation/` — controllers บางเฉียบ มีแค่ input validation + routing
    - `setup/ioc/` — DI container wiring

  - **ใช้ Dishka แทน FastAPI Depends** — เพราะ FastAPI Depends ทำให้ "coupling กับ framework" ซึ่งขัดกับ Clean Architecture ที่มอง framework เป็น replaceable outer layer Dishka เป็น DI container ที่ framework-agnostic ทำให้แต่ละ layer ไม่จำเป็นต้องรู้จัก FastAPI

  - **Pydantic Caution — อย่าใช้ Pydantic ใน Domain Layer** — Pydantic models ใน presentation ต้องแยกจาก business layer โดยสิ้นเชิง Business validation ไม่ควรใช้ Pydantic เพราะมันช้ากว่า dataclass และสร้าง coupling กับ external library แนะนำให้ใช้ dataclass สำหรับ request data และ TypedDict สำหรับ response data

  - **Adapters ได้รับข้อยกเว้นจาก Dependency Rule** — ผู้เขียนยอมรับว่าในทางปฏิบัติ Infra และ Presentation layers ต้องพึ่งพาทั้ง inner layers และ external components "retains nearly all the advantages of Clean Architecture while simplifying real-world development"

  - **TOML-based Configuration** — ใช้ TOML เป็น source of truth สำหรับ settings (ไม่ใช่ .env) รองรับหลาย environment (local, dev, prod)

  - **Hierarchical RBAC** — 3 roles: user, admin, super admin (ต้อง grant ผ่าน DB โดยตรง) พร้อม policy ที่ว่า super admin จะถูก soft-delete หรือเปลี่ยนสิทธิ์ผ่าน API ไม่ได้

- **นำไปใช้ยังไง:** เหมาะสำหรับโปรเจกต์ที่ต้องการ Clean Architecture แบบจริงจัง — โดยเฉพาะแนวคิด Dishka DI และ Pydantic Caution ที่เป็น pain point จริงของทีมที่ใช้ FastAPI

### 4. n1nj4t4nuk1/python-ddd-skeleton

- **คืออะไร:** Skeleton/Template สำหรับ REST API Python ที่ใช้ Hexagonal Architecture (Ports & Adapters), Domain-Driven Design, CQRS และ Event Sourcing อิงจาก CodelyTV typescript-ddd-skeleton ที่มีชื่อเสียง

- **สิ่งที่ได้:**

  - **Bounded Context เป็นโครงสร้างหลัก** — จัด code ภายใต้ `src/contexts/shared/` ซึ่งเป็นหัวใจของ DDD: ระบบถูกแบ่งเป็น bounded contexts แต่ละ context มี ubiquitous language และโมเดลของตัวเอง ส่วน `shared` เป็น shared kernel สำหรับ value objects/base classes ที่ใช้ร่วมกัน

  - **Hexagonal Architecture + CQRS + Event Sourcing — ครบชุด** — DDD triad ที่ทรงพลัง: Hexagonal แยก core domain ออกจาก infrastructure via port/adapter, CQRS แยก command/query models, Event Sourcing เก็บ state changes เป็น stream ของ domain events แทน current state snapshot

  - **SOLID Principles เป็นแกนหลัก** — ทุกการออกแบบคลาสและ interface อิง SOLID ทำให้ domain layer ยืดหยุ่น เปลี่ยน infrastructure ได้โดยไม่กระทบ business logic

  - **Framework-agnostic โดยธรรมชาติ** — FastAPI เป็นแค่ delivery mechanism (inbound adapter) ซึ่งสอดคล้องกับแนวคิด Hexagonal Architecture ที่ framework เป็นแค่ "ทางเข้า" ที่เปลี่ยนได้

  - **เหมาะเป็น Template ไม่ใช่ Framework** — ผู้เขียนจงใจทำให้เป็น template ที่ขยายได้ ไม่ใช่ framework ตายตัว มี 146 commits แสดงถึงการพัฒนาต่อเนื่อง

- **นำไปใช้ยังไง:** ใช้เป็นแนวทางออกแบบระบบขนาดใหญ่ที่ต้องแยก bounded contexts ชัดเจน — โดยเฉพาะส่วน shared kernel และ event sourcing pattern ที่สามารถนำมาปรับใช้กับระบบที่ต้องการ audit log และ event-driven architecture
